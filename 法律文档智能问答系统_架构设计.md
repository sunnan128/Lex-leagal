## 各层对比：规划 vs 实际

### 1. 接入层

| 组件     | 规划架构               | 实际实现                | 选型理由                                     |
| ------ | ------------------ | ------------------- | ---------------------------------------- |
| 前端界面   | Streamlit / Gradio | **Streamlit**       | 开发效率高，Python 原生，Demo 阶段够用。Gradio 更适合 ML 演示，Streamlit 更适合数据应用。支持多文件上传、异步进度轮询、实时进度条显示 |
| API 网关 | FastAPI            | **FastAPI** ✅ 按规划实现 | 异步高并发，自动生成 Swagger 文档，调试方便               |
| 用户管理   | JWT 认证 + 请求日志      | **未实现**             | 本地使用阶段不需要。计划在云部署时加上 Nginx basic auth 作为最小安全方案，长期考虑 JWT |

### 2. 数据层

| 组件       | 规划架构             | 实际实现                               | 选型理由                                     |
| -------- | ---------------- | ---------------------------------- | ---------------------------------------- |
| 文档上传     | LangChain Loader | **PyMuPDF + python-docx**          | 直接调用更轻量，不引入 LangChain 全量依赖。PyMuPDF 解析速度比 LangChain PDFLoader 快 3~5 倍 |
| 文档切分     | 递归字符切分 + 按条款切分   | **按条款正则切分 + MIN_CHUNK_SIZE=80 合并** | 法律文档结构特殊，"第X条" 是天然边界，比固定字数切分更语义完整。加入最小合并阈值避免过碎 |
| NER 实体抽取 | 命名实体识别           | **未实现**                            | 对当前"检索→问答"核心链路不是必须的。后续作为增强功能，可抽取当事人、案由、法条引用等信息用于结构化检索 |

> `PyMuPDF` 和 `python-docx` 是强大的**解析引擎**，LangChain 的加载器则是基于这些引擎（及其他库）打造的、为LLM应用提供标准化接入的**高级工具**。
>
> 选择哪一个，取决于你的需求：
>
> - 如果追求在LangChain项目中**高效、标准化地加载数据**，那么**LangChain加载器**是首选。
> - 如果需要进行**深度、定制化的文档解析**，那么直接使用 **PyMuPDF 或 python-docx** 会更合适

### 3. 检索层（核心亮点）

| 检索路径   | 规划架构                  | 实际实现                         | 选型理由                                     |
| ------ | --------------------- | ---------------------------- | ---------------------------------------- |
| 语义向量检索 | Milvus + bge-large-zh | **ChromaDB** + 同款模型          | ChromaDB 零运维、直接嵌入 Python 进程，单机场景足够。Milvus 适合分布式/海量数据，当前阶段过度设计 |
| 关键词检索  | BM25                  | **BM25** ✅ 按规划实现             | 法律场景必需——用户会搜索"民法典第584条"这种精确条款号，向量检索对精确匹配不稳定 |
| 查询归一化  | —                     | **阿拉伯→中文数字** ✅ 新增优化          | 用户输入"第100条"但文档写"第一百条"，自动转换查询中的数字格式，BM25 匹配率从 0% 提升到 100%。覆盖条/款/章/节四种单位，对用户无感知 |
| 图谱检索   | Neo4j Cypher          | **未实现**                      | 需要预定义的实体关系 schema 和有监督的关系抽取模型，投入较大。当前以文档级检索为主 |
| 重排序    | bge-reranker-large    | **bge-reranker-large** ✅ 已实现 | 混合检索取 top-20 → CrossEncoder 精排 → top-5 给 LLM。懒加载模块级单例，不阻塞启动。large 精度更高，首次下载后本地缓存，推理 20 对 ~500ms |

**混合召回策略**：语义检索权重 0.7 + 关键词权重 0.3，合并后按总分排序取 top-K。检索前自动对查询做**数字格式归一化**（"第100条"→"第一百条"），解决用户习惯输入阿拉伯数字但文档使用中文数字的不匹配问题。

**ChromaDB vs Milvus 决策树**：

```
数据量 < 100 万条？ ──Yes──→ ChromaDB（零运维，嵌入应用）
       │
       No
       ↓
需要分布式/高可用？ ──Yes──→ Milvus（集群部署）
       │
       No
       ↓
    ChromaDB（单机够用）
```

### 4. 生成层

| 组件       | 规划架构                    | 实际实现                            | 选型理由                                     |
| -------- | ----------------------- | ------------------------------- | ---------------------------------------- |
| LLM 推理   | DeepSeek / Qwen（Ollama） | **DeepSeek Chat（API）**          | 效果优于 7B 本地模型，API 调用成本极低（百万 token ~1 元）。保留 Ollama 接口作为离线备用 |
| Agent 编排 | LangChain + LangGraph   | **直接 FastAPI 编排**               | 当前单轮问答场景不需要 Agent 的复杂编排逻辑，直接调用更简洁、更容易调试  |
| 引用溯源     | 检索结果附带出处                | **文档名 + 页码 + 段落号 + 内容片段** ✅ 已实现 | 核心功能，每份回答都附带引用卡片，可追溯到原文精确位置              |

**Prompt 安全设计**：

```
规则 1：只基于提供的文档内容回答
规则 2：文档中没有相关信息 → 明确回答"知识库中未找到相关信息"
规则 3：引用时标注文档来源
```

### 5. 存储层

| 数据库   | 规划架构   | 实际实现                          | 选型理由                                     |
| ----- | ------ | ----------------------------- | ---------------------------------------- |
| 结构化存储 | MySQL  | **未实现**                       | 当前无用户管理需求，SQLite 轻量够用。未来用户量上来再迁移 MySQL   |
| 向量存储  | Milvus | **ChromaDB PersistentClient** | 胜在简单：`pip install chromadb` 即可，数据自动持久化到磁盘，API 直观 |
| 图数据库  | Neo4j  | **未实现**                       | 知识图谱构建成本高，需要标注数据和关系抽取模型。作为二期工程规划         |

### 6. 部署

| 维度   | 规划架构          | 实际实现              | 选型理由                                     |
| ---- | ------------- | ----------------- | ---------------------------------------- |
| 部署方式 | Docker + 云服务器 | **本地运行 + 部署方案文档** | 代码和 Dockerfile 已就绪，实际上线需要云服务器资源。配套 4 套部署方案供选择 |
| 前端运行 | Gradio        | **Streamlit**     | 同前端选型理由                                  |

### 7. 评估层（新增）

| 组件         | 规划架构     | 实际实现                                   | 选型理由                                     |

## 定稿的架构决策记录（ADR）

### ADR-1：用 ChromaDB 而非 Milvus

- **背景**：原架构选型 Milvus，但实际选择 ChromaDB
- **决策**：使用 ChromaDB PersistentClient
- **理由**：单机场景、数据量 < 100 万条、零运维、嵌入进程
- **代价**：不支持分布式，未来扩展需迁移

### ADR-2：用 DeepSeek API 而非本地模型

- **背景**：原架构计划用 Ollama 部署本地 7B 模型
- **决策**：使用 DeepSeek Chat（OpenAI 兼容接口）
- **理由**：效果远超 7B 本地模型，成本极低（百万 token ~1 元），接口标准易切换
- **代价**：依赖网络，离线不可用（保留 Ollama 切换能力）

### ADR-3：延迟知识图谱（Neo4j）到二期

- **背景**：原架构包含 Neo4j 知识图谱
- **决策**：当前版本不实现，规划到 v2.0
- **理由**：知识图谱构建需要标注数据 + 关系抽取模型，投入产出比在起步阶段不高
- **代价**：无法做"当事人→案件→法条"的关系推理

### ADR-4：直接编排而非 LangChain Agent

- **背景**：原架构用 LangChain + LangGraph 做 Agent 编排
- **决策**：直接在 FastAPI 中编排检索→生成流程
- **理由**：当前单轮问答不需要 Agent 的多步推理，直接调用更简洁、调试更容易
- **代价**：多轮对话和复杂推理需要额外开发

### ADR-5：RAGAS 自动化评估 + LLM-as-a-Judge 降级

- **背景**：每一项优化（Rerank、查询归一化等）的效果评估全凭"感觉"，缺乏量化基准
- **决策**：构建 30 条 QA 对的评估数据集（10 领域）× RAGAS 自动化流水线 × A/B 对比报告
- **理由**：RAGAS（faithfulness / answer_relevancy / context_precision / context_recall）是工业级 RAG 评估标准，未安装时自动降级到 LLM-as-a-Judge（知识库覆盖率/平均处理时间/引用数等替代指标），零额外依赖
- **代价**：如果文档内容更新，需要同步更新数据集中的 ground_truth；评估耗时 3~5 分钟/30条
