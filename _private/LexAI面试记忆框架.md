# LexAI 面试记忆框架

> 核心思路：不要背细节，记"故事线"和"关键决策"。面试时沿着故事线讲，细节自然带出来。
> 
> 最后更新：加入 Rerank 重排序（优化#17）

---

## 一句话介绍（电梯演讲，背这个就行）

> "LexAI是一个法律领域的RAG问答系统。它解决的核心问题是：大模型在法律场景下会编造法条，这是致命的。所以我的系统只基于检索到的法律文档回答，每条回答都能追溯到原文出处。技术上用了语义检索+BM25混合召回+Rerank精排的三阶段检索架构，17个迭代优化让系统从Demo变成了可部署的产品。"

---

## 故事线：面试时按这个顺序讲

```
痛点 → 方案 → 架构 → 难点 → 优化 → 部署 → 反思
```

每一步只需要记住一个关键词，细节自然会想起来。

### 第1步：痛点（30秒）

**记忆锚点：法律不允许说错**

- 大模型幻觉在法律场景是致命的——编造法条可能导致错误决策
- 法律条文需要精确引用——"第584条"不是"大概第580多条"
- 法律频繁更新——Fine-tune一次成本高，不如RAG随时更新知识库

### 第2步：方案（30秒）

**记忆锚点：让AI从"编造者"变成"引用者"**

三个核心设计决策：
1. **检索增强生成（RAG）**：LLM只基于检索结果回答，不检索就拒绝回答
2. **三阶段检索**：混合检索（粗排）→ Rerank精排（细排）→ LLM生成，工业级标准流程
3. **引用溯源**：每条回答附带文档名+页码+段落号

### 第3步：架构（1分钟）

**记忆锚点：6层，每层一个关键词**

| 层 | 关键词 | 你做了什么 |
|---|---|---|
| 接入层 | **Streamlit + FastAPI** | 前端用Streamlit快速搭Demo，后端FastAPI提供REST接口 |
| 数据层 | **按条款切分** | 法律文档按"第X条/款/章/节"正则切分，不是按固定字数 |
| 检索层 | **三阶段检索** | ChromaDB向量(0.7)+BM25(0.3)混合粗排 → bge-reranker-base精排 → top-5给LLM |
| 生成层 | **DeepSeek + 溯源** | LLM生成回答，附带引用卡片 |
| 知识层 | **BGE嵌入 + Rerank** | bge-large-zh-v1.5做嵌入，bge-reranker-base做精排，均支持降级加载 |
| 存储层 | **ChromaDB持久化** | ChromaDB PersistentClient，数据自动存磁盘；BM25用pickle持久化 |

**面试技巧**：不用背每一层的细节，只要能说出"我选了什么、为什么选它"就够。

### 第4步：难点（重点，面试必问）

**记忆锚点：5个真实问题**

不用背代码，记住"遇到了什么→怎么解决的→效果":

#### 难点1：上传速度慢
- **问题**：法律文档按条款切分后产生上百个小片段，每个都要过编码模型，上传一个193KB文档要很久
- **方案**：MIN_CHUNK_SIZE=80，把小于80字的相邻片段合并
- **效果**：编码次数减少3-5倍，上传速度提升3-5倍

#### 难点2：模型下载不了
- **问题**：国内连不上Hugging Face，服务直接崩
- **方案**：6层降级策略——本地缓存→ModelScope→HF镜像→轻量模型→极小模型
- **效果**：已缓存秒级加载，未缓存自动走国内源，不会崩

#### 难点3：重启后文档丢失/删不掉
- **问题**：FastAPI热重载清空内存缓存，文档列表消失；恢复的文档删不掉
- **方案**：从ChromaDB元数据恢复文档列表 + 同步写入内存缓存 + 兜底直接走ChromaDB删除
- **效果**：重启无感知，删除正常

#### 难点4：BM25内存占用
- **问题**：文档量增长后BM25占内存线性增长，重启需要全量重建索引
- **方案**：pickle磁盘持久化，加载时直接反序列化
- **效果**：启动速度从5-10秒降到0.1秒

#### 难点5：混合检索不够精确（Rerank优化）
- **问题**：加权融合是浅层融合，无法精确建模query和文档的深层语义关系，最相关的可能排到第3-5位
- **方案**：引入bge-reranker-base CrossEncoder精排，混合检索取top-20候选→精排→top-5给LLM
- **效果**：排序依据从加权分数变成深度语义相关性，top-5命中率显著提升
- **技术决策**：选base不选large（推理快3-5倍，精度损失~1%）；CrossEncoder不阻塞启动（懒加载单例）；加载失败自动降级回退

### 第5步：亮点功能（加分项）

**记忆锚点：4个超出基本RAG的功能**

1. **Rerank三阶段检索**：混合检索粗排 → CrossEncoder精排 → LLM生成，工业级RAG标准流程
2. **异步上传+实时进度条**：后台线程处理上传，前端0.5秒轮询，用户能看到解析→编码→存储各阶段进度
3. **文档预览双模式跳转**：按片段号直接跳转 / 按原段落号自动跨页搜索+高亮
4. **引用卡片直达定位**：点击"查看原文"自动跳到引用段落并高亮

### 第6步：部署（展示工程能力）

**记忆锚点：4套方案覆盖全场景**

| 方案 | 成本 | 场景 |
|---|---|---|
| 内网共享 | 0 | 小团队办公室 |
| Tailscale组网 | 免费 | 远程办公 |
| 云服务器 | ~50元/月 | 7×24正式上线 |
| Docker | ~50元/月 | 环境隔离 |

关键技术点：Nginx反向代理 + HTTPS(Let's Encrypt) + systemd守护进程 + 基本认证

### 第7步：架构决策（展示思考深度）

**记忆锚点：5个ADR（架构决策记录）**

面试官最喜欢问"为什么选X不选Y"。这5个决策你一定要能说清楚：

| # | 决策 | 选了什么 | 为什么 |
|---|---|---|---|
| 1 | 向量数据库 | ChromaDB 而非 Milvus | 单机<100万条，零运维，嵌入进程 |
| 2 | LLM | DeepSeek API 而非本地7B | 效果远超7B，百万token~1元，接口标准 |
| 3 | 知识图谱 | 延迟到v2.0 | 需要标注数据+关系抽取模型，起步阶段投入产出比不高 |
| 4 | Agent编排 | 直接FastAPI 而非 LangChain | 单轮问答不需要Agent，直接调用更简洁易调试 |
| 5 | Rerank模型 | bge-reranker-base 而非 large | 推理快3-5倍，20对~200ms，精度损失~1%，适合实时场景 |

## 按岗位的讲解策略

### 面试 RAG 工程师（重点讲检索层）

核心故事线：

```
混合检索设计 → Rerank精排 → 文档切分策略 → 检索质量优化 → 评估方法
```

必说：
- **三阶段检索架构**（这是核心升级）：粗排（混合检索top-20）→ 精排（CrossEncoder top-5）→ 生成
- 为什么需要混合检索（向量检索擅长语义匹配，但"第584条"这种精确匹配不稳定）
- 为什么需要Rerank（加权融合是浅层排序，CrossEncoder能建模query-doc深层交互）
- 为什么选CrossEncoder不选BiEncoder（精度更高，BiEncoder虽可预编码但无法捕捉交互）
- 文档切分为什么按条款不按字数（法律文档的"第X条"是天然语义边界）
- 权重怎么定的（0.7+0.3，实验得出的，语义为主关键词为辅）
- BM25持久化解决了什么（启动速度和内存问题）

### 面试 Agent 工程师（重点讲生成层+可扩展性）

核心故事线：

```
当前直接编排 → 为什么不用Agent → 未来怎么加Agent → LangGraph规划
```

必说：
- 当前单轮问答不需要Agent（ADR-4的决策理由）
- 如果要做多轮对话，会用LangGraph的StateGraph管理对话状态
- 如果要做复杂查询（比如"这个合同有哪些风险点"+"帮我生成修改建议"），可以拆成多个Agent：检索Agent→分析Agent→生成Agent
- 课程中学的MCP协议怎么用在未来的扩展中

## 记忆方法总结

**不要背：** 代码细节、具体参数值、每一个优化记录

**要记住：**
1. 一句话介绍（30秒讲清楚项目是什么）
2. 痛点→方案→架构的故事线（按顺序讲，不卡壳）
3. 5个难点（每个难点一句话概括）
4. 5个架构决策（为什么选X不选Y）
5. 按岗位的侧重点（RAG讲三阶段检索，Agent讲可扩展性）

**练习方法：** 对着镜子讲3遍"介绍一下你的项目"，每次控制在2分钟内。讲顺了，面试就不会卡。
