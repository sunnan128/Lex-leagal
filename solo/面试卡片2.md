# 面试卡片 2 — 多智能体企业知识问答系统

---

## 卡片 0：项目一句话（电梯演讲）

> 我做了一个多智能体企业知识问答系统。员工提问后，**路由 Agent** 用 LLM 做意图分类，知识问答走**检索 Agent**、操作请求走**执行 Agent**。检索 Agent 用 **BM25 + 向量混合检索**，**RRF 融合**后用 **Reranker 精排**，RAGAS 评估显示精准率达到 **91%**。执行 Agent 通过 **MCP 协议**调用运维工具。最后**审查 Agent** 用 **Guardrails** 做格式校验和幻觉检测，不通过就打回重试，最多 2 次。整个链路用 **LangGraph** 编排，**FastAPI** 对外提供服务。

---

## 卡片 1：整体架构（必问）

### 核心设计
- 用 **LangGraph** 编排 **4 个 Agent**：路由 → 检索 → 执行 → 审查

### 为什么是 4 个而不是 1 个
- 单一 Agent 做太多事，prompt 会膨胀
- 职责分离更好调试、更好维护
- 每个 Agent 专注一件事，输出更稳定

### 状态传递
- 用 **TypedDict** 在 Agent 间传递状态
- **不是全局变量**，链路清晰可追溯

---

## 卡片 2：路由 Agent（高频）

### 为什么不用 if-else，而用 LLM 分类
- 用户表达方式多样
- "网挂了"和"网络故障"是同一个意图
- LLM 理解语义，if-else 只能硬匹配

### confidence 阈值设 0.6 的原因
- 低于 0.6 说明模型不确定
- 保守走**人工**更安全，避免误判

---

## 卡片 3：RAG 检索链路（核心考点）

### 做了什么
- **BM25 + 向量双路召回**
- **RRF 融合**后接 **Reranker 精排**

### 为什么这么做
| 组件 | 解决的问题 |
|------|-----------|
| BM25 | 纯向量检索对"VPN-001服务器"这类专有名词召回率低 |
| 向量检索 | 补充语义匹配能力 |
| RRF | 无需训练的融合算法，简单有效 |
| Reranker (cross-encoder) | 弥补 bi-encoder 的语义鸿沟，精排结果 |

### 踩了什么坑
1. **jieba 分词对 IT 术语分不准** → 加了自定义词典
2. **RRF 的 k 值**一开始用默认 60 → 测试后 k=40 效果更好
3. **Reranker 逐条打分太慢** → 改成批量处理

### 数据 / 指标（重点记）
| 阶段 | 指标 |
|------|------|
| 纯向量检索 Top-5 召回率 | **68%** |
| 混合检索 Top-5 召回率 | **84%** |
| 加 Reranker 后 Top-5 精准率 | **91%** |

## 卡片 4：Agent 编排（高频）

### 审查不通过为什么要打回路？由，而不是原地重试
- 可能是**路由本身分错了**
- 重新走完整链路更可靠
- 原地重试只会重复同样的错误

### 最多重试 2 次的原因
- 防止**死循环**
- 2 次还过不了，说明系统确实处理不了
- 此时应该走人工兜底

---

## 卡片 5：Guardrails（亮点）

### 面试官问"输出不可控怎么办"的标准答案

三层防护：

```
格式校验 → 幻觉检测 → 敏感信息过滤
```

| 层级 | 作用 |
|------|------|
| **格式校验** | 检查输出是否符合要求的 JSON/格式模板 |
| **幻觉检测** | 对比检索结果，检测模型是否编造内容 |
| **敏感信息过滤** | 过滤密码、密钥、个人隐私等敏感数据 |

---

## 卡片 6：降级策略（加分）

### 三级降级的设计逻辑

> 每一级都比上一级更"笨"，但更可靠

```
Level 1（正常）:  完整链路 → 4个Agent全跑
Level 2（降级）:  跳过Reranker → 只用混合检索
Level 3（保底）:  直接返回检索结果 → 不走LLM生成
```

- **Level 1 → Level 2**：Reranker 超时或失败
- **Level 2 → Level 3**：LLM 服务不可用
- 保证任何情况下都有结果返回，服务不挂

---

## 卡片 7：评估体系（区分度）

### 核心思想

> 不是"我觉得效果好"，而是 **"RAGAS 数据表明 faithfulness 从 X 提升到 Y"**

### 评估维度
| 指标 | 含义 |
|------|------|
| **faithfulness** | 回答是否忠实于检索到的文档 |
| **answer_relevancy** | 回答是否与问题相关 |
| **context_precision** | 检索到的文档是否精确 |
| **context_recall** | 是否检索到了所有相关内容 |

### 怎么用评估驱动优化
- RAGAS 评估发现检索问题 → **调整分块策略**
- faithfulness 低 → 检查 prompt 是否约束不够
- 用数据说话，而不是凭感觉
