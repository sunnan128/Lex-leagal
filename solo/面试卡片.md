# LexAI 面试背诵卡片

## 一、项目一句话介绍

> 基于 RAG 架构的法律文档智能问答系统，支持**溯源回答**，**杜绝幻觉**。

---

## 二、核心痛点与解决方案（必背）

| 痛点 | 解决方案 |
|------|----------|
| 大模型幻觉（编造法条） | 只基于检索到的文档回答，检索不到明确告知"未找到" |
| 回答无法溯源 | 引用溯源卡片：文档名 + 页码 + 段落号 + 内容快照 |
| 条款号精确检索 | 混合检索：语义检索 + BM25 关键词（如"第584条"精确命中） |
| 知识更新滞后 | 上传新文档即扩充知识库，无需重新训练 |

---

## 三、技术架构（按层说）

```
接入层：FastAPI（后端API）+ Streamlit（前端界面）
数据层：PyMuPDF + python-docx（文档解析）
检索层：ChromaDB（向量检索）+ BM25（关键词检索）
生成层：OpenAI API / Ollama 本地部署
存储层：ChromaDB 向量存储 + SQLite
```

---

## 四、技术亮点（加分项）

### 1. 混合检索策略
- **语义向量检索**：BGE 模型，处理语义匹配
- **关键词检索**：BM25，处理精确条款号匹配
- **结果融合**：权重合并后重新排序

### 2. 幻觉抑制
- 严格基于检索结果回答
- 检索不到时明确声明"知识库中未找到相关信息"
- 所有回答附带引用来源
- LLM 温度参数设为 0.3（降低随机性）

### 3. 法律文档适配
- 按"第X条"、"第X款"等标识智能切分
- 保留文档结构信息
- 记录页码和段落号用于溯源

### 4. 工程优化
- **懒加载**：CrossEncoder 首次查询时才加载，不阻塞启动
- **单例缓存**：模块级全局变量，避免重复加载到显存
- **优雅降级**：模型加载失败时自动回退到纯混合检索
- **异步上传**：后端异步处理，前端轮询进度条

## 六、面试高频问题

### Q1：为什么用 RAG 而不是直接微调模型？
> 法律领域要求**准确性**和**可溯源**，微调模型仍然可能幻觉，且更新成本高。RAG 让模型只引用原文，不编造内容，新文档上传即可扩充知识库。

### Q2：怎么解决检索不到但模型乱答的问题？
> 两层防护：①检索结果为空时直接返回"未找到"；② prompt 中明确指示"只能基于提供的参考资料回答，不要编造"。

### Q3：BM25 和向量检索有什么区别？什么时候用哪个？
> BM25 擅长精确关键词匹配（如"第584条"），向量检索擅长语义匹配（如"合同违约怎么办"）。两者结合互补。

### Q4：如果模型服务挂了怎么办？
> 优雅降级设计，CrossEncoder 加载失败时自动回退到纯混合检索，保证核心问答功能可用。

### Q5：怎么部署的？
> 支持三种方式：① Python 脚本启动；② Docker Compose 一键部署；③ 手动分别启动前后端。
