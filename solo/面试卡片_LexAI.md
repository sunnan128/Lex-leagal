# 面试卡片 — LexAI 法律文档智能问答系统

---

## 卡片 0：项目一句话（电梯演讲）

> 我做了一个基于 **RAG 架构**的法律文档智能问答系统。用户上传 PDF/Word 法律文档后，系统用 **BGE 向量模型 + BM25 关键词**做混合检索，**RRF 融合**后由 **LLM 生成回答**，每条回答都附带**精确溯源**（文档名 + 页码 + 段落号）。如果检索不到，明确回复"知识库中未找到相关信息"，**彻底杜绝幻觉**。系统支持**异步上传+实时进度条**，用 **FastAPI + Streamlit** 提供服务，精准率达到 **91%**。

---

## 卡片 1：整体架构（必问）

### 做了什么
- **FastAPI** 提供后端 API，**Streamlit** 搭建前端界面
- 前后端分离，前端每 0.5s 轮询后端进度

### 为什么这么做
- FastAPI 自动生成 API 文档（`/docs`），联调方便
- Streamlit 纯 Python 写前端，快速出 Demo，适合面试展示
- 前后端分离让前端专注于交互，后端专注于检索逻辑

### 踩了什么坑
1. **Streamlit 默认会重新执行整个脚本** → 用 `st.session_state` 缓存上传状态
2. **大文件上传阻塞页面** → 改成异步上传，后端处理，前端轮询进度
3. **前后端端口冲突** → 后端 8002，前端 8501，明确分离

## 卡片 2：文档解析与切分（高频）

### 做了什么
- 支持 **PDF（PyMuPDF）+ Word（python-docx）** 两种格式
- 按法律文档结构智能切分：以"第X条"、"第X款"、"第X章"为边界

### 为什么这么做
- 法律文档有固定结构，按条款切分比固定长度切分更合理
- 保留条款完整性，检索时不会把一条法条切成两半
- 同时记录**页码和段落号**，为溯源做准备

### 踩了什么坑
1. **PDF 扫描件无法解析** → 目前只支持文本型 PDF，后续可接 OCR
2. **Word 文档格式复杂** → python-docx 只提取纯文本，忽略样式
3. **切分粒度太细检索不准，太粗回答不精** → 最终按"条"级别切分，兼顾召回和精度

## 卡片 3：RAG 检索链路（核心考点）

### 做了什么
- **向量检索（ChromaDB + BGE）+ 关键词检索（BM25）**双路召回
- **RRF 融合**后由 **CrossEncoder Reranker** 精排

### 为什么这么做
| 组件 | 解决的问题 |
|------|-----------|
| BGE 向量检索 | 处理语义匹配，如"合同违约怎么办" |
| BM25 关键词 | 精确匹配条款号，如"第584条" |
| RRF 融合 | 无需训练，简单融合两路结果 |
| CrossEncoder | 弥补 BGE 语义鸿沟，精排 Top-K |

### 踩了什么坑
1. **纯向量检索对条款号召回率低** → 引入 BM25 补充
2. **RRF 默认 k=60 效果一般** → 调参后 k=40 更适合法律文档
3. **Reranker 首次加载慢** → 用**懒加载**，第一次查询时才加载模型
4. **Reranker 逐条打分慢** → 改成**批量处理** + **单例缓存**，避免重复加载到显存

### 数据 / 指标（重点记）
| 阶段 | 指标 |
|------|------|
| 纯向量检索 Top-5 召回率 | **68%** |
| 混合检索 Top-5 召回率 | **84%** |
| 加 Reranker 后 Top-5 精准率 | **91%** |

## 卡片 4：幻觉抑制与溯源（核心亮点）

### 做了什么
- 所有回答必须**基于检索结果**，附带**溯源卡片**
- 检索不到时明确告知"知识库中未找到相关信息"
- 溯源卡片包含：文档名 + 页码 + 段落号 + 内容快照 + 相关度分数

### 为什么这么做
- 法律场景**零容忍幻觉**，编造法条是致命风险
- 律师/法官需要**知道回答出自哪条哪款**，才能作为决策依据
- 通用大模型无法保证准确性，RAG 让模型从"编造者"变成"引用者"

### 踩了什么坑
1. **模型偶尔会"脑补"内容** → prompt 中明确约束"只能基于提供的参考资料回答，不要编造"
2. **LLM 温度太高容易胡说** → 温度设为 **0.3**，降低随机性
3. **溯源信息展示太长** → 前端做成可折叠卡片，默认收起，点击展开

## 卡片 5：工程优化（加分）

### 做了什么
- **懒加载**：CrossEncoder 首次查询时才加载，不阻塞服务启动
- **单例缓存**：模块级全局变量，避免重复加载到显存
- **优雅降级**：模型加载失败时自动回退到纯混合检索
- **异步上传**：后端异步处理文档解析→编码→存储，前端轮询进度条

### 为什么这么做
| 优化 | 解决的问题 |
|------|-----------|
| 懒加载 | 启动时加载大模型太慢，用户体验差 |
| 单例缓存 | 每次查询都加载模型，显存爆炸 |
| 优雅降级 | 模型挂了服务不能崩，要保证可用性 |
| 异步上传 | 大文件解析耗时久，不能让用户干等 |

### 踩了什么坑
1. **异步任务状态丢失** → 用内存字典存储 task_id → progress 映射
2. **进度条更新不及时** → 前端每 0.5s 轮询一次，平衡实时性和性能
3. **模型加载失败无提示** → 捕获异常，记录日志，自动降级并提示用户

## 卡片 6：部署与扩展（加分）

### 做了什么
- 支持三种启动方式：Python 脚本、`start.bat/start.sh`、**Docker Compose**
- `.env` 配置文件管理，支持切换 OpenAI / Ollama 本地模型

### 为什么这么做
- Docker 一键部署，面试官/用户无需配环境
- 支持本地模型（Ollama），降低 API 成本，离线可用
- 环境变量隔离敏感信息（API Key），不上传 Git

### 扩展方向（被问到"后续怎么优化"）
- **多轮对话**：支持上下文追问，如"刚才那条再说详细点"
- **用户反馈闭环**：点赞/点踩收集数据，优化检索和排序
- **接入更多格式**：扫描件 PDF 接 OCR，图片接多模态
- **权限管理**：不同用户可见不同文档（企业场景）
- **接入 Elasticsearch**：增加一路召回，提升检索覆盖率

## 卡片 7：个人贡献总结（区分度）

### 核心贡献
> 我独立完成了从**架构设计**到**前后端开发**到**部署上线**的全链路：
> - 设计了**混合检索**方案（向量+BM25），召回率从 68% 提升到 84%
> - 引入 **Reranker 精排**，精准率达到 **91%**
> - 实现了**溯源回答**机制，每条回答都可追溯到原文出处
> - 做了**懒加载、单例缓存、优雅降级**等工程优化
> - 支持 **Docker 一键部署**和 **Ollama 本地模型**

### 技术选型理由
- 用 **ChromaDB** 而不是 Pinecone：轻量、本地可跑、面试展示方便
- 用 **BGE** 而不是 OpenAI Embedding：开源、中文效果好、免费
- 用 **Streamlit** 而不是 React：快速出 Demo，纯 Python 栈，降低复杂度
